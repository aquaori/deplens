# Deplens 面试题库（面试表达版）

这份题库不是写给“看代码的人”的，而是写给“在面试里要把项目讲清楚的人”的。

写法原则：
- 每道题都尽量按“背景 -> 方案 -> 原理 -> 取舍”来回答
- 表达尽量口语化，方便你直接复述
- 回答会尽量说明“为什么这么做”，而不是只说“做了什么”

---

## 一、项目定位与整体设计

### 1. 你这个项目是做什么的？
答：这是一个 Node.js 依赖分析工具，核心目标是帮助开发者判断一个项目里哪些依赖真的在被使用，哪些依赖没有被使用，哪些依赖被代码引用了但却没有在 `package.json` 里声明。它最开始主要解决的是传统单项目的依赖审计问题，后面我把它扩展到了 monorepo、多包管理器、证据链、AI review、预复核这些能力。

### 2. 你最初为什么想做这个项目？
答：因为很多前端和 Node 项目随着时间推移，会逐渐积累很多历史依赖。有些依赖其实已经不用了，有些依赖是幽灵依赖，有些是因为脚本、配置、构建链间接使用，看起来没 import 但其实不能删。市面上的一些工具能做基础检查，但对 monorepo、workspace、pnpm、工具链场景支持不够，所以我想做一个更偏工程实战的依赖治理工具。

### 3. 这个项目和 depcheck 这类工具最大的区别是什么？
答：我觉得最大的区别有四点。第一，它不只是扫源码 import，还会结合 lockfile、workspace、evidence 做更完整的判断。第二，它支持 monorepo，不是只看单包。第三，它能输出证据链，不只是给一个结论。第四，我后面又接了 AI review，让它可以对低置信的依赖做二次复核，这一点是传统工具通常没有的。

### 4. 你是怎么定义“高质量依赖分析”的？
答：我觉得不是“报出越多 unused dependency 越厉害”，而是要同时做到三件事。第一，结论尽量准。第二，用户能看懂为什么这么判。第三，复杂场景下能告诉用户哪里有不确定性，需要人工看一眼。所以我后来专门补了 evidence、signals、candidate 分级和 AI 复核，就是为了把“准确性”和“可解释性”同时拉起来。

### 5. 你这个项目的核心架构是什么？
答：我会把它分成六层。第一层是传统静态分析层，负责扫文件、解析 AST、找 import 和 require。第二层是 evidence 层，把声明、引用、问题整理成结构化证据。第三层是 query/domain 层，把这些证据变成可调用接口。第四层是 monorepo 聚合层，解决 workspace 和多包问题。第五层是 AI review 层，用 LangChain 做二次复核。第六层是 CLI / TUI 展示层，负责把结果以 check 或 review 的形式呈现给用户。

### 6. 这个项目最重要的设计原则是什么？
答：静态分析主导，AI 做补充，而不是相反。也就是说，确定性事实一定由程序去算，比如 import、require、package.json、lockfile、workspace 关系；AI 只负责处理那些传统分析拿不准的部分，比如工具链依赖、字符串形式依赖、配置文件依赖、低置信候选复核。这样系统才稳，不会变成“全靠模型猜”。

### 7. 你觉得这个项目最有工程价值的点是什么？
答：我觉得不是“接了大模型”这件事本身，而是我把它从一个只会输出结果的分析器，逐步做成了一个“可解释、可复核、可交互”的依赖治理助手。也就是说，用户不仅知道结果，还能知道为什么、证据在哪、哪些是高置信、哪些需要人工判断。

### 8. 这个项目现在主要面向哪些场景？
答：第一种是平时做依赖清理，比如排查 unused dependency。第二种是 monorepo 依赖治理，比如 workspace 之间的依赖声明问题。第三种是 code review 辅助，比如判断某个依赖到底能不能删。第四种是依赖风险排查，比如 ghost dependency、构建链隐式依赖、脚本依赖这些。

### 9. 为什么后来要把项目往 Agent 方向演进？
答：因为静态分析很擅长找确定性事实，但它不擅长解释复杂场景。比如某个依赖没有标准 import，却可能通过 Babel、Vite、scripts 或 config 间接使用。这种时候用户最关心的不是一个冷冰冰的结论，而是“为什么”“能不能删”“证据在哪”。所以我引入 Agent，不是为了追热点，而是为了补上解释和复核这一层。

### 10. 你会怎么一句话概括这个项目？
答：这是一个以静态分析为核心、以证据链为骨架、以 AI 复核为增强层的 Node / monorepo 依赖治理工具。

## 二、扫描、AST 与基础静态分析

### 11. 你最开始是怎么做依赖分析的？
答：最开始就是比较经典的一套流程：先扫描项目源码文件，再把源码解析成 AST，然后遍历 AST，提取 `import`、`require`、`import()` 这类依赖引用；最后再把这些引用和 `package.json` 里的声明依赖做对比，得出哪些依赖被使用了，哪些依赖没有被使用。

### 12. 为什么要用 AST，而不是简单正则搜索 import？
答：因为正则只能做很粗的字符串匹配，很容易误判。AST 的好处是它能理解语法结构，知道一个字符串到底是 import 路径、普通字符串、配置项，还是别的东西。比如 `require("lodash")`、动态 import、JSX、TS 语法、成员表达式，这些都需要语法级理解。AST 的稳定性和准确性会比正则高很多。

### 13. 你的 AST 是怎么解析出来的？
答：我用的是 Babel parser。原因很现实，它对 JS、TS、JSX 这些语法支持比较全，而且扩展 plugin 比较方便。项目里会有 TS、JSX、顶层 await、import attributes 这些语法，所以我在 parser 层做了兼容处理。

### 14. 为什么不用 TypeScript Compiler API？
答：不是说不能用，而是当时从落地成本和语法兼容性来看，Babel parser 更合适。这个项目最核心的需求是“提取依赖引用”，不是做完整类型分析，所以 Babel parser 在这个场景下更轻、更实用，而且处理 JSX、现代语法时也比较顺手。

### 15. 你具体会在 AST 里关注哪些节点？
答：核心关注几类。第一是 `ImportDeclaration`，也就是标准 ES import。第二是 `CallExpression` 里形如 `require("dep")` 的调用。第三是 `ImportExpression`，也就是动态 import。后面为了做 signals，我还会看 `StringLiteral`、`TemplateLiteral` 这些字符串节点，以及一些特殊调用模式，比如 `require.resolve("dep")`。

### 16. 动态 import 为什么会成为一个问题？
答：因为动态 import 有时参数不是静态字符串，而是变量拼接、模板字符串或者运行时生成的，这种情况下传统静态分析没法 100% 判断目标依赖到底是什么。所以最开始这些情况会导致不确定性，后来我在输出里把它们和普通静态引用区分开了，避免误导用户。

### 17. 这个项目在扫描文件时会遇到哪些现实问题？
答：现实问题很多。比如有些项目里会混杂 TS、JSX、mjs、cjs、测试文件、配置文件、fixture。还有一些文件可能有 BOM，或者包含新语法、实验语法。再比如 monorepo 下子包很多、路径深、workspace pattern 很复杂，所以扫描和解析要做很多兼容。

### 18. 你后来为什么专门修了 import attributes 和 deprecated assert 相关问题？
答：因为我在分析 Vite 仓库时碰到了真实问题。Vite 里面有一些文件使用了旧的 `assert` 风格 import attributes，而 Babel 这块新旧语法切换比较敏感。如果 parser plugin 开错，扫描阶段就会报错，直接影响依赖分析的完整性。后来我做了回退策略，先按新语法试，失败时再按旧 assert 语法重试。

### 19. 你为什么还写了 transpiler 这一层？
答：因为项目不只是想看原始源码，有些情况下需要把 TS/JSX 之类的代码转成更标准的 JS 再做进一步处理。比如某些依赖使用线索可能藏在转译相关流程里，另外也方便统一后续解析。这个层本身后来还成为一个典型案例，因为 Babel preset/plugin 依赖就是通过这里的字符串和构建链被间接使用的。

### 20. 你怎么处理 parser 和 transpiler 的语法兼容问题？
答：我的思路是不要假设所有项目都只用一种语法，而是做“正常解析 + 特定错误回退”。像 import attributes 那类问题，我不是一次性开所有 plugin，而是按错误类型选择回退，这样更稳，也不会互相冲突。

## 三、依赖判定与基础结果生成

### 21. 你怎么判定一个依赖是否未使用？
答：最基础的判定逻辑是：这个依赖在 `package.json` 里声明了，但在项目源码中没有发现对应的静态引用，也没有被其它确定性证据命中，那它就会进入 unused dependency 列表。后来为了减少误报，我又把“没有被标准引用，但可能通过 config/tooling/scripts 使用”的情况单独分层，不再一刀切。

### 22. 你怎么判定 ghost dependency？
答：ghost dependency 的定义是：代码里出现了对某个依赖的引用，但当前包的 `package.json` 没有声明它。这个场景在 monorepo 里很常见，尤其是 hoist 或 workspace 关系复杂时，代码跑得起来，但依赖声明其实不规范。

### 23. 为什么 lockfile 对这个项目很重要？
答：因为只看源码和 `package.json` 还不够。lockfile 能补出很多安装层面的事实，比如到底装了哪个版本、pnpm importer 对应哪个 package、workspace link 是怎么解析的。尤其 monorepo 和 pnpm 场景下，不结合 lockfile 很难把依赖关系看清。

### 24. 你为什么没有一开始就把 devDependencies 一起强分析？
答：因为 devDependencies 的使用语义比较复杂。很多依赖不是在业务代码里 import，而是在构建、测试、脚本、配置里使用。如果一开始就把 devDependencies 按业务依赖的标准去分析，会产生很多误报。所以我的策略是先把基础分析做稳，再逐步补 config、scripts、signals 这些增强层。

### 25. 你怎么过滤 Node 内置模块？
答：像 `fs`、`path`、`node:fs` 这类模块，本质上不是 npm 依赖，所以不能混进 ghost 或 unused 判定里。我在 evidence/query 层里把这类模块排除了，避免把它们当成第三方依赖。

### 26. 你之前为什么会出现 `undefined` 这种脏输出？
答：那是因为动态 import 一类的结果和普通依赖名混在一起了，渲染层在没有做清洗的情况下直接把空名字打印出来，就变成了 `undefined`。后来我把动态项单独分开，输出时不再混进普通依赖列表。

### 27. 为什么 BOM 会影响 package.json 解析？
答：因为 UTF-8 BOM 在编辑器里通常看不见，但对 `JSON.parse` 来说它就是一个真实字符。如果字符串前面带了 BOM，又没有先去掉，`JSON.parse` 就可能报错。Vite 的一个 fixture 正好就用这个场景测 UTF-8 BOM，所以我后来专门做了去 BOM 的 JSON 读取逻辑。

### 28. 你后面为什么改成自动识别包管理器，而不是让用户传 `--pnpm`？
答：因为让用户自己决定包管理器，既增加心智负担，也容易选错。尤其 monorepo 里，子包甚至可能不共享同一种安装上下文。后来我改成自动检测最近的 lockfile 和 `packageManager`，让系统自己决定走 npm 还是 pnpm，这样结果更准确，也更符合真实项目使用习惯。

### 29. 自动包管理器识别的原理是什么？
答：简单说就是从目标目录往上找最近的 lockfile，比如 `package-lock.json` 或 `pnpm-lock.yaml`，再结合 `packageManager` 字段和 workspace 结构来决定 driver。这样单项目和 monorepo 子包都可以自动选择最合适的解析方式。

### 30. 你最初的分析结果结构长什么样？
答：最初主要是一些统计和列表，比如：
- 总依赖数
- 已使用依赖
- 未使用依赖
- ghost dependency
- 动态 import 信息

后来随着 evidence、signals、review 加入，结果不再只是“列表”，而是变成了一个更完整的结构化 report。

## 四、monorepo 分析原理

### 31. 你是怎么识别一个项目是 monorepo 的？
答：主要看两类线索。第一类是 `package.json` 里的 `workspaces` 字段。第二类是 `pnpm-workspace.yaml` 这类 workspace 配置文件。只要命中这些特征，我就会把项目从单包分析切换到 monorepo 分析流程。

### 32. monorepo 分析的基本思路是什么？
答：思路是把每个 workspace package 当成一个相对独立的 Node 项目来分析，但最终又把它们的结果聚合成一个全局视角。这样既能看到单包问题，也能看到整个 monorepo 的依赖治理状态。

### 33. 你怎么找到所有子包？
答：我会读取 workspace pattern，然后按 glob 展开，去找每个子包的 `package.json`。找到 manifest 后，再以它为入口做包级分析。

### 34. monorepo 里为什么不能简单复用单项目逻辑？
答：因为 monorepo 有几个额外复杂点。第一是 workspace 包之间互相依赖。第二是 lockfile 可能在根目录，不在子包目录。第三是 pnpm 的 importer / workspace link / hoisting 机制会影响依赖解析。第四是很多“能跑起来”的依赖关系其实是不规范的，尤其会暴露 ghost dependency 和 undeclared workspace dependency。

### 35. 你怎么处理 monorepo 下 lockfile 不在子包目录的问题？
答：这是后来做自动包管理器识别时重点解决的问题。不是要求每个子包自己目录里有 lockfile，而是从子包位置向上找最近的 lockfile，再结合 importer 去定位它在 lockfile 中对应的那一段信息。这样结果才真实。

### 36. 你怎么聚合 monorepo 的包级结果？
答：我做了根级 report 和根级 evidenceIndex。report 层聚合统计，比如包数量、哪些包有问题、哪些包问题最多；evidenceIndex 层聚合声明、引用、issue、signals 这些证据，方便后面跨包查询和 AI 使用。

### 37. monorepo 输出为什么后来改成“只详细展示有问题的包”？
答：因为真实 monorepo 可能有几百个包，如果每个包都逐个打印“没有问题”，输出会非常噪。后来我改成默认只把重点包单独展开，其余没有问题的包做汇总展示，这样更适合人类阅读。

### 38. monorepo 分析里最典型的问题是什么？
答：我觉得最典型的是三类。第一类是 undeclared workspace dependency，也就是包之间用了但没声明。第二类是 ghost dependency，特别是在 hoist 场景里很容易出现。第三类是测试、playground、fixture 导致的依赖声明混乱。

### 39. 你怎么看 monorepo 对依赖分析的挑战？
答：最大的挑战不是“包多”，而是“边界模糊”。单包项目里，依赖关系通常比较直接；monorepo 里，根包、子包、workspace link、hoist、脚本、配置都会搅在一起。如果没有根级聚合视角，只看单包很容易看不清真实问题。

### 40. 你为什么说 monorepo 是这个项目的一个亮点？
答：因为很多依赖分析工具在单包场景能工作，但一到 monorepo 就容易失真。我的项目后面把：
- workspace 发现
- lockfile 自动定位
- 根级 evidence 聚合
- 子包独立分析
- workspace 依赖问题输出

这几件事都串起来了，所以 monorepo 不再只是“能跑”，而是能真正分析出有价值的问题。

## 五、evidence 证据链层

### 41. 你为什么要做 evidence 层？
答：因为只输出结果不够。用户和面试官都会问：你为什么说它 unused？为什么说它是 ghost？证据在哪？如果系统答不上来，它就只是一个黑盒分析器。所以我做 evidence 层，本质上是把程序的推理依据显式化。

### 42. DeclarationEvidence 的作用是什么？
答：它回答的是“这个依赖在什么地方被声明了”。比如它在哪个 `package.json`、属于 `dependencies` 还是 `devDependencies`、版本范围是什么。没有这一层，你只能知道“项目里有这个依赖”，但说不清它从哪里来。

### 43. ReferenceEvidence 的作用是什么？
答：它回答的是“这个依赖在什么地方被使用了”。比如文件路径、行列、引用类型。没有这一层，你只能说“被用了”，但说不清“在哪用的、怎么用的”。

### 44. IssueEvidence 的作用是什么？
答：它回答的是“为什么系统把它判成某种问题”。比如为什么它是 `unused-dependency`，为什么它是 `ghost-dependency`。这能把最终问题和底层证据连接起来。

### 45. evidence 和 report 的关系是什么？
答：report 更像“汇总结果”，evidence 更像“底层证据明细”。report 适合直接展示给用户看，而 evidence 适合给 query、Agent、review 或更深入的解释功能用。

### 46. 你为什么说 evidence 比直接把 AST 给 AI 更有价值？
答：因为 AST 太底层、太碎，而且很多节点对回答问题其实没有直接价值。evidence 是“已经整理过的事实”，比如声明、引用、问题、signals，这样 AI 不用重新理解整棵 AST，就能直接基于高价值结构化信息工作。

### 47. 你在 evidence/query 层做了什么？
答：我做的是“底层证据查询”。也就是围绕 evidence 提供一些查证接口，比如：
- 查声明
- 查引用
- 查 issue
- 查 signal
- 给出某个依赖的完整证据解释

这一步相当于把 evidence 变成了程序可调用的检索能力。

### 48. monorepo 里的 evidenceIndex 是什么？
答：它是 monorepo 根级证据索引，相当于把所有子包的 declarations、references、issues、signals 聚合到一起。这样就能支持跨包查询，比如“这个依赖在整个 monorepo 里出现在哪些包里”“哪些包都声明了它”“哪些包对它有问题”。

### 49. evidence 为什么是后面 AI 接入的前提？
答：因为如果没有 evidence，AI 拿到的只有最终列表，很难做可靠复核；或者只能直接读全仓代码，成本又太高。有了 evidence，AI 至少先有一层确定性摘要，再决定要不要进一步获取代码上下文。

### 50. evidence 层给这个项目带来的最大变化是什么？
答：最大的变化是，系统从“会出结果”变成了“会给理由”。这不只是用户体验更好，也让后面的 query API、review 模式、AI 复核、局部代码复核都有了稳定的基础。

## 六、signals 与非标准依赖线索

### 51. 你为什么会想到引入 signals？
答：因为我发现传统静态分析只能稳定抓标准 import/require，但很多真实依赖使用方式根本不长那样。比如 Babel preset/plugin 是字符串，脚本命令不在源码里 import，config 文件里也可能通过约定式写法间接依赖。所以我需要一层“线索证据”来补这些盲区。

### 52. SignalEvidence 的定位是什么？
答：它不是最终结论，而是“弱证据”。也就是说，signal 只能说明“这里有可疑线索”，不能直接说明“这个依赖一定被用了”或者“一定没被用”。它的作用是给后面的 candidate 分级和 AI 复核提供输入。

### 53. 目前支持哪些 signalType？
答：目前主要有四类。`string-literal` 表示普通字符串命中。`tooling-string` 表示这个字符串更像出现在构建链或工具链语境里。`require-resolve` 表示类似 `require.resolve("dep")` 的使用线索。`script-command` 表示这个依赖作为命令出现在 `package.json scripts` 中。

### 54. 你是怎么采集字符串类 signal 的？
答：我在 AST 遍历时会看 `StringLiteral` 和一些字符串相关节点，如果字符串内容长得像依赖名，而且它不是标准 import/require 引用，就把它记成 signal。这样像 Babel plugin 名、preset 名、loader 名这些字符串就不会被完全忽略。

### 55. tooling-string 和普通 string-literal 有什么区别？
答：区别在于上下文。如果一个字符串只是出现在普通源码里，那只能算普通 `string-literal`。但如果它出现在明显的 tooling 语境里，比如 Babel、transform、plugin、loader、preset、resolve 这些上下文中，我会把它升级成 `tooling-string`。这会让它在后续 candidate 分级里权重更高。

### 56. fileRole 是干什么的？
答：fileRole 是给命中的文件打角色标签。比如这个 signal 是来自业务源码、配置文件、工具链文件、脚本文件，还是测试文件。它很重要，因为同样一个依赖名，出现在 `src/` 和出现在 `tailwind.config.ts`、`transpiler.ts` 的意义是完全不同的。

### 57. 你怎么判断一个文件更像 tooling 而不是 source？
答：我会结合文件路径和命中的上下文判断。比如文件名、目录名里如果出现 `transpiler`、`bundler`、`loader`、`plugin`、`preset`、`tooling`、`build` 这些词，我会更倾向把它归到 tooling。另外如果字符串附近的对象 key、函数调用链里有 Babel、transform、plugin 等语境，也会增强这种判断。

### 58. script-command signal 的价值是什么？
答：它主要解决脚本依赖问题。像 `nodemon`、`vite`、`tsx`、`eslint` 这些依赖可能完全不在业务源码里 import，但它们确实在 scripts 里使用。如果只看源码就会误判，这时 script-command 就能给出“它至少在脚本里有使用线索”。

### 59. signals 和 evidence 的关系是什么？
答：signals 也是 evidence 的一种，只不过它属于“弱证据”而不是“确定性引用证据”。我后来把它也挂进了 evidence 体系，就是为了让 query、review、Agent 都能统一消费，而不是单独维护一套线索逻辑。

### 60. signals 给项目带来的最大提升是什么？
答：它让系统第一次能“看见”那些非标准依赖使用方式。虽然还不能完全准确判断，但至少不再是完全盲的。这是后面 candidate 分级和 AI 二次复核能成立的前提。

## 七、candidate 分级与 AI 复核入口

### 61. 你为什么不直接让 AI 去复核所有依赖？
答：因为那样成本太高，速度也太慢，而且很多依赖其实根本不需要 AI 参与。比如已经有明确引用的依赖，程序就能确定它被使用了，再拿去复核纯属浪费 token。所以我先做了 candidate 分级，用规则筛掉那些确定性场景，只把低置信候选交给 AI。

### 62. confirmed-used 是什么含义？
答：就是这个依赖已经有足够强的静态证据证明它被用了，比如标准 import、require、明确的引用证据。对于这种依赖，后续复核就没有必要再做。

### 63. high-confidence-unused 是什么含义？
答：就是这个依赖没有标准引用，也没有 signal，没有 config/script/tooling 线索，程序侧已经比较有把握认为它没有被使用。对于这类依赖，通常不需要 AI 参与，直接作为高置信 unused 输出即可。

### 64. likely-tooling-usage 是什么含义？
答：它表示“虽然没有标准业务引用，但从 signals 看，它很像是通过 config、tooling、构建链、脚本间接使用”。这类依赖典型地包括 Babel plugin/preset、构建插件、脚本工具等。它们不应该被简单当成“可直接删除的 unused dependency”。

### 65. needs-review 是什么含义？
答：意思是“有一些线索，但程序还不能稳稳地下结论”。比如有字符串命中，但上下文不够强；或者有一些脚本/config 线索，但还不能确认是不是合法使用。这种就是最适合交给 AI 二次复核的对象。

### 66. ghost-dependency 为什么也可以进入 candidate 体系？
答：因为 ghost 的表象虽然是“引用了但没声明”，但有些场景里可能涉及 alias、virtual module、fixture、workspace 约定，静态分析未必能完全判断它到底是不是标准 npm 依赖。所以低置信 ghost 也可以进入复核体系，只不过目前我更重点处理的是 unused 的误报问题。

### 67. candidate 分级的核心价值是什么？
答：它最大的价值是“控成本”和“提效率”。一方面，AI 不需要碰所有依赖，只处理最值得处理的那一小部分。另一方面，系统可以把最终结果分成高置信和低置信两类，不会把所有结论说得一样肯定。

### 68. 这层分级是规则驱动还是 AI 驱动？
答：这层是规则驱动。AI 还没介入。我的思路一直是先用程序做第一层初筛，再让 AI 处理模糊情况。这样系统更稳，也更可控。

### 69. 为什么 preReview 后来只复核静态 unused 里的可疑对象？
答：因为这是一个很重要的成本优化。既然 `confirmed-used` 这些依赖已经被静态分析排除了，那它们就没必要再拿去复查。真正有价值的，是那些已经落入 `unusedDependencies` 列表、但又带有 signals 的对象，也就是“看起来没用，但可能有误报”的这部分。

### 70. 这套 candidate 机制和 AI 结合后最大的优势是什么？
答：它把 AI 变成了一个“有闸门、有边界的复核器”，而不是一个全局乱分析的黑盒。你可以很清楚地解释：为什么这个依赖会被送去 AI 复核，为什么另一个不会，这样系统的工程逻辑是闭环的。

## 八、LangChain 在项目中的作用

### 71. 你为什么选择用 LangChain？
答：因为我需要的不是一个简单的模型 SDK，而是一个能够把“模型 + 工具调用 + 提示词约束 + 运行时”组织起来的框架。LangChain 对这种场景比较合适，尤其是我已经有很多 query/domain 接口，只需要把它们包装成 tools 给模型用。

### 72. LangChain 在这个项目里到底扮演什么角色？
答：它不是核心分析引擎，核心分析还是我自己的 analyzer、evidence、query。LangChain 在这里更像“编排层”，负责把模型、工具、提示词和 review 会话组织起来。也就是说，静态分析是引擎，LangChain 是让引擎变成 Agent 的胶水层。

### 73. 你为什么没一上来就用 LangGraph？
答：因为项目当前阶段的需求还没有复杂到必须上图工作流。现阶段主要是单 Agent + tool calling + 结构化输出，LangChain 已经够用了。LangGraph 更适合特别复杂的多步骤状态机和多 Agent 协作，我当时判断没必要一开始就上那么重的框架。

### 74. 你是怎么接 Qwen API 的？
答：因为 Qwen 提供了 OpenAI-compatible API，所以我直接用 `@langchain/openai` 去接，通过传 `model`、`apiKey` 和 `baseURL` 就能跑起来。这样不需要专门换另一套模型客户端，工程上更省事。

### 75. 你在项目里定义了哪些 LangChain tools？
答：主要有项目摘要、包摘要、unused/ghost 查询、依赖概览、删除评估、review candidate 查询、candidate 二次复核、代码上下文 bundle 查询这些工具。它们本质上都是对现有 query/domain 能力的封装。

### 76. 这些 tools 和你原来的 query/domain 接口是什么关系？
答：tools 不是新发明的一套能力，而是把已有能力暴露给 AI。query/domain 是“程序内部的业务接口”，tools 是“给模型调用的入口”。这样我就不会把逻辑写两遍。

### 77. 你怎么看待 Prompt 在这个项目里的作用？
答：Prompt 很重要，但我一直没有把它当成唯一手段。我的理解是：Prompt 负责告诉模型身份、边界、工具使用方式和输出规范，但真正决定结果质量的还是底层证据、工具和结构化输入。Prompt 是约束层，不是魔法。

### 78. 为什么你后面又做了结构化输出，而不是只靠 Prompt 让模型“别输出 markdown”？
答：因为纯 Prompt 约束并不可靠。模型天然偏好用 markdown、标题、列表来组织内容，特别是在做总结和计划时。后来我改成要求它输出固定 JSON schema，再由前台渲染。这样展示控制权就回到程序手里了。

### 79. LangChain 帮你实现了哪些以前不好做的能力？
答：最直接的是 tool calling，也就是模型可以根据问题自己决定要不要查项目摘要、依赖概览、review candidate 或代码上下文。第二是把 review 模式做成自然语言入口，不再只靠死命令。第三是让 AI 能参与二次复核，而不是纯文本聊天。

### 80. 你会怎么评价 LangChain 在这个项目里的价值？
答：我会说它不是让项目“从 0 变成 1”，而是让项目“从 1 变成更像产品”。因为 1 的部分还是我自己的静态分析系统，LangChain 的价值是在现有能力之上，加了一层可交互、可复核、可工具调用的智能入口。

## 九、review 命令全过程与原理

### 81. review 命令的完整流程是什么？
答：大体上是五步。第一，检查 AI 配置是否存在。第二，对目标项目先跑一次静态分析，拿到结构化 report。第三，用 report 构造 review runtime，包括模型、工具、提示词和会话上下文。第四，启动交互式界面。第五，用户提问后，模型按需调用工具，再把结果渲染成终端可读的输出。

### 82. review 为什么要先跑一次静态分析？
答：因为 review 不是纯聊天，它的价值来自“基于当前项目真实分析结果回答问题”。如果没有这一步，模型只能凭常识回答。先跑静态分析，相当于先准备一份项目快照，后续多轮对话都围绕这份快照展开。

### 83. review 的 runtime 里主要有什么？
答：主要有四样东西。第一是模型对象。第二是工具集合，也就是对 query/domain 能力的封装。第三是 system prompt，用来约束模型身份、工具使用和输出格式。第四是当前项目的 report 以及可选的 enhancement 结果，作为会话上下文。

### 84. review 模式为什么不是每问一句都重新扫描？
答：因为那样太慢，也没必要。用户进入 review 的本质是围绕同一份项目快照做多轮问答，所以正确做法是：启动时分析一次，后续会话都复用这份结果，除非用户主动要求重新扫描。

### 85. review 里的问题类型大致怎么分？
答：大概可以分成几类。第一类是摘要型，比如“这个项目整体依赖情况怎么样”。第二类是查询型，比如“有哪些未使用依赖”。第三类是解释型，比如“为什么这个依赖看起来没用”。第四类是决策型，比如“它能不能删”。第五类是 monorepo 相关问题，比如“哪个子包问题最多”。

### 86. review 里模型是怎么知道该调哪个 tool 的？
答：靠 system prompt 里的工具说明和问题类型引导。比如问“能不能删”，模型会倾向调用 `can_remove_dependency`、`get_dependency_review_candidate`、`review_dependency_candidate`。问“哪里用到了”，它就会调用 `get_dependency_context_bundle` 或 `get_dependency_overview`。

### 87. review 为什么后面还做了 fallback 兜底策略？
答：因为并不是所有问题都能靠现有工具稳定回答。有些问题太宽泛，或者工具链不足，模型可能会来回试探，甚至递归过深。所以我设计了 fallback：当系统判断问题复杂、成本可能上升时，提示用户是继续深度分析、补充问题还是中止回答。

### 88. 你为什么还专门做了 review 的 TUI？
答：因为如果只是命令行一问一答，体验会比较像脚本，而不是一个真正的交互式助手。TUI 能提供欢迎页、聊天区、状态栏、输入区、滚动、状态提示，这样更像一个完整产品，也更适合后续扩展。

### 89. review 模式里为什么要做结构化输出渲染？
答：因为纯自然语言对 CLI 来说不稳定，尤其模型很容易输出 markdown、长段文字、表格、复杂列表。后来我让模型输出固定结构，再由前台按 `dependency_list`、`plan`、`assessment`、`code_context` 等类型渲染，终端效果就稳定很多。

### 90. review 模式最关键的工程价值是什么？
答：我觉得不是“可以聊天”，而是“能围绕一份真实分析快照做多轮、可追问、可复核的依赖治理对话”。这比单纯的 `check` 更适合复杂问题和人工审查场景。

## 十、review 的提示词工程与输出渲染

### 91. 你是怎么设计 review 的 system prompt 的？
答：我会把 prompt 分成几块来设计。第一块是身份设定，明确它是 Deplens 依赖分析助手，不是泛用聊天机器人。第二块是工具使用规则，强调 tool first、证据优先。第三块是风险约束，不能编造不存在的命令或功能。第四块是输出协议，要求返回结构化 JSON。

### 92. 为什么你后来特别强调“工具优先”？
答：因为如果不强调这一点，模型很容易用自己的常识直接回答，这样就会偏离项目真实情况。我的目标不是让它像百科全书，而是像一个基于当前仓库事实工作的助手。所以必须要求它优先调用工具、优先信任工具返回的数据。

### 93. 你为什么还要求模型“不要虚构不存在的功能”？
答：因为真实测试时，模型可能会编出类似 `deplens check --fix` 这种项目里根本没有的命令。对工程工具来说，这种幻觉很危险，所以我在 prompt 里明确要求：除非工具结果或已知功能里明确存在，否则不要臆造命令和能力。

### 94. 你为什么不再完全信任模型自己排版？
答：因为模型排版的稳定性太差，特别是在终端环境里。它一会儿用 markdown，一会儿用标题，一会儿又套表格。后来我把“布局决定权”从模型手里拿回来，让它只负责产出结构，前台负责渲染，这样视觉效果和可读性都更可控。

### 95. 你现在支持哪些结构化回答类型？
答：主要有五类。`text` 用于普通文本说明。`dependency_list` 用于依赖列表。`plan` 用于治理计划。`assessment` 用于风险判断和结论。`code_context` 用于展示代码片段或证据上下文。这几类基本覆盖了当前 review 的常见场景。

### 96. code_context 类型的意义是什么？
答：它让 review 不再只会说“在某某文件某某行”，而是可以把相关代码片段更清楚地展示出来。对于“这个依赖为什么可能还在用”这类问题，这种展示会比纯文本解释更有说服力。

### 97. 你是怎么处理模型不按 schema 输出的情况的？
答：我做了 fallback。也就是说，优先尝试解析 `<deplens_json>...</deplens_json>` 里的 JSON，如果失败，就退回 plain text fallback，再做一层简单清洗和兜底渲染。这样即使模型偶尔不守规矩，系统也不会直接崩。

### 98. 为什么你后来感觉模型回答“更冷静”了？
答：因为约束更强了。结构化输出、工具优先、禁止幻觉、禁止 markdown、终端友好这些规则一起上，会显著压低模型自由发挥的空间。它不是真的变笨了，而是变得更克制、更像工程助手了。

### 99. 如果面试官说“这不就是 prompt engineering 吗”，你怎么回答？
答：我会说 prompt 只是最外层。真正让结果靠谱的是下面几层：静态分析、evidence、signals、query/domain、candidate 分级、tool calling、结构化输出和渲染。Prompt 只是把这些能力组织起来，并约束模型怎么使用它们，不是核心能力本身。

### 100. review 这套设计给用户体验带来的最大提升是什么？
答：我觉得是把“冷工具”变成了“能解释、能追问、能复核的助手”。用户不再只是看到一堆 unused dependency，而是能继续问：为什么是它？能不能删？有没有证据？是不是误报？这就是产品体验上的差别。

## 十一、preReview 全过程与原理

### 101. preReview 是什么？
答：它是一个“分析增强模式”。意思是在传统静态分析结束后，再让 AI 对低置信候选做一轮二次复核，最终给出更细的分层结果，而不是只停留在粗筛级别的 unused 列表。

### 102. 你为什么会想到做 preReview？
答：因为我发现纯静态分析在一些场景下容易误报，尤其是构建链、config、scripts、tooling 这些依赖。用户如果直接根据粗筛结果删依赖，可能会把项目删挂。所以我想增加一层 AI 复核，帮系统在复杂场景下再多看一眼。

### 103. preReview 为什么后来改成显式开关？
答：因为一开始我做过默认预复核，但实际问题很明显：启动太慢、token 消耗太高、在大型 monorepo 里尤其夸张。用户也不一定每次都值得花这个成本。所以后来我改成只有显式 `--preReview` 时才启用。

### 104. preReview 现在具体复核哪些对象？
答：现在只复核“静态分析已经判成 unused，但同时又带有可疑 signals 的依赖”。换句话说，它不是全量扫所有依赖，也不是扫所有 candidate，而是只针对“看起来没用，但可能误报”的那一小部分。

### 105. 为什么不复核已经被静态分析判定为已使用的依赖？
答：因为那是浪费 token。像 `@babel/types`、`@langchain/core` 这类已经有明确静态引用的依赖，程序已经能确认它们在被使用了，再交给 AI 复核没有收益，只会徒增成本。

### 106. preReview 的完整流程是什么？
答：第一步，先跑静态分析。第二步，构建 signals 和 candidate。第三步，从静态 unused 中筛出可疑对象。第四步，对这些对象调用 AI 做二次复核。第五步，把复核结果整合进最终输出。也就是说，它不是一个完全独立的新流程，而是分析流程的增强段。

### 107. preReview 和 review 的关系是什么？
答：preReview 更像“批量增强分析”，适合在 check 里提前把结果分层。review 更像“交互式追问和解释”，适合用户围绕某个依赖继续深挖。所以它们都用到了 AI，但定位不一样。

### 108. preReview 最终输出为什么要改成分层结果？
答：因为如果还是只输出原始 unused 列表，那 AI 复核就没有真正融入结果。后来我把输出改成：
- 高置信未使用
- 疑似间接/工具链使用
- 仍需人工复核

这样用户一眼就知道哪些真的可以优先清理，哪些不能直接删。

### 109. 你为什么把 extra info 放到最下面？
答：因为用户最关心的是结论，而不是附加说明。之前 extra info 插在中间，会打断主结果的阅读流。后来我调整成：先看分析结果，再看 preReview 提示，最后再看 extra info，这样更符合阅读顺序。

### 110. preReview 这套设计最能体现你什么工程思路？
答：我觉得体现了“把 AI 放在合适的位置，而不是默认开最大功率”。也就是：先让程序做能做的事，再让 AI 只补程序的盲区，而且还要受成本和场景约束。这是一个比较现实的工程思路。

## 十二、AI 配置、安全与成本控制

### 111. 为什么你后来加了 API 配置检查？
答：因为如果用户没配 `QWEN_API_KEY` 之类的环境变量，而系统又直接进入 AI 流程，最后会在 LangChain 初始化或者调用模型时抛出一串很难懂的错误。我的做法是提前检查，没配就直接拒绝执行 `review` 或 `check --preReview`，给出明确提示。

### 112. 你会检查哪些环境变量？
答：主要是三个：`QWEN_MODEL`、`QWEN_API_KEY` 和 `QWEN_BASE_URL`。这三个缺任何一个，review 或 preReview 的 AI 流程都没法正常运行。

### 113. 为什么普通 check 不要求配置 AI？
答：因为普通 check 本质上还是传统静态分析，不依赖模型。这样做的好处是用户即使没有 API Key，也仍然能正常使用这个工具最核心的分析能力。AI 是增强层，不是前置必备条件。

### 114. 你怎么控制 token 成本？
答：主要靠三层。第一层是 candidate 分级，把大部分确定性场景挡在 AI 外面。第二层是 preReview 默认关闭，只在显式开启时运行。第三层是局部上下文而不是全仓输入，也就是只让 AI 看相关依赖、相关 signals、相关代码片段，而不是整仓乱喂。

### 115. 你怎么平衡“准确性”和“成本”？
答：我的原则是：静态分析先做主判定，AI 只补低置信盲区。也就是说，不为了追求 100% 覆盖而让所有场景都走 AI，而是先把 80% 的确定性问题用程序解决，再把 20% 的复杂问题交给 AI 处理。这样成本和收益比较平衡。

### 116. 为什么你认为 review 默认不应该全局预复核？
答：因为 review 是交互式场景，很多时候用户只是想快速看一下项目整体情况，不一定需要一上来就把所有低置信依赖都过一遍 AI。如果默认全局预复核，启动会很慢，成本也很高，而且用户未必真的需要那么深。

### 117. 你为什么给复杂问题设计了 fallback 交互？
答：因为复杂问题不是不能答，而是可能需要更高成本。与其让系统静默烧 token 或长时间卡着，不如明确告诉用户：这个问题比较复杂，继续分析可能会消耗更多 token，你是要继续、补充信息，还是取消。这种交互更诚实，也更符合产品体验。

### 118. 如果面试官问“你怎么防止 AI 幻觉”，你怎么答？
答：我会说我没有把所有事情交给模型，而是通过几层手段压幻觉。第一，工具优先，模型先读程序算出来的事实。第二，结构化输出，减少模型自由发挥。第三，禁止虚构不存在的命令和能力。第四，复杂问题要么走局部代码复核，要么提示用户继续深度分析，不让模型无依据乱猜。

### 119. 你为什么说“静态分析主导，AI 补盲区”是更安全的？
答：因为依赖分析很多结论其实是确定性的，比如某个 import 就在那里，某个 package.json 就声明了这个依赖。这类事实最适合程序算，不适合让模型猜。模型真正有价值的是在这些事实之外做解释、整合和低置信复核。这样系统会更稳，也更容易讲清楚原理。

### 120. 如果用户没配 API，但是执行了 review 或 check --preReview，系统应该怎么做？
答：应该直接拒绝，并明确告诉用户缺少哪些环境变量，为什么需要这些配置，以及哪些命令依赖 AI。这样用户一眼就知道问题在哪，不会被内部异常信息搞晕。

## 十三、代码复核、局部上下文与“准 RAG”

### 121. 你现在的 review / preReview 是 RAG 吗？
答：严格说，目前还不是完整意义上的向量检索 RAG。现在做的是“证据链驱动的 AI 复核 + 局部代码上下文增强”。也就是说，AI 主要还是先看 candidate、overview、signals，再按需要补局部代码片段，而不是先把全仓做成向量库再语义检索。

### 122. 你为什么没有一上来就做全仓 RAG？
答：因为那样很重，成本也不稳定。我的项目已经有很强的静态结构信息了，比如 evidence、signals、candidate，这些都能天然缩小范围。所以更合理的做法是：先用静态结果定位可疑依赖，再围绕这个依赖去定向获取代码上下文，而不是让模型一开始就去全仓搜索。

### 123. 你后来做的代码复核功能，核心原理是什么？
答：核心原理是“signals 定位 + 定向读文件 + 局部代码片段 bundle + AI 复核”。也就是：先根据 references 和 signals 找到相关文件和位置，再从这些位置附近截取局部代码片段，最后把这些片段交给 AI 结合证据链一起判断。

### 124. 为什么这种局部代码复核比只给文件路径和行号更有价值？
答：因为文件路径和行号对程序有意义，但对人和模型来说还是太抽象了。给一个具体代码片段，模型才能理解这个依赖到底是以插件名字符串出现、还是普通字符串、还是某个配置数组的一部分。对用户来说，看到代码也更容易信服。

### 125. 你具体是怎么做 context bundle 的？
答：我新增了 `CodeContextSnippet` 和 `DependencyContextBundle`。然后在 `getDependencyContextBundle()` 里，根据某个依赖的 `references` 和 `signals` 去定位文件，读取文件内容，按焦点行往前后截若干行，做成带行号、带焦点标记的 snippet，最后汇总成一个 bundle 返回给 Agent。

### 126. 这个局部上下文 bundle 在 AI 复核里怎么用？
答：在 `runSecondaryReview()` 里，模型的输入不再只是 candidate 和 overview，还会加上 context bundle。这样 AI 做二次复核时，不仅知道“有 signal”，还知道这个 signal 旁边到底是什么代码。比如它能看出这是 Babel transform 的 plugin 配置，而不是普通业务代码里的某个字符串。

### 127. 为什么你把它叫“准 RAG”而不直接叫 RAG？
答：因为它的思路已经有 RAG 味道了，也就是“先定位，再喂局部上下文”，但它还没有真正上向量索引、embedding 和 retriever。它目前更像一个静态结果驱动的局部检索系统，所以我会叫它“局部上下文增强”或者“准 RAG”。

### 128. 如果以后真的要接 RAG，你会怎么做？
答：我不会一开始把整个 AST 或整个仓库直接塞进知识库，而是优先把：
- signal 命中的代码片段
- config 文件片段
- scripts
- evidence 邻近代码
- package manifests

这些内容切成文档块，再按 dependencyName、packageName、fileType 做 metadata 过滤，最后做局部检索。也就是说，会做“依赖中心化的局部 RAG”，而不是全仓大锅炖。

### 129. 局部 RAG 对项目的提升会体现在哪？
答：主要体现在解释能力上。比如现在系统可以说“这个依赖在 `src/utils/transpiler.ts` 某一行附近出现了 tooling-string”，以后加局部 RAG 后，它可以进一步说“这是在 Babel transform 配置中，以插件字符串的形式传入，因此属于构建链合法使用，不建议删除”。也就是从“定位”升级到“上下文解释”。

### 130. 你会怎么向面试官解释“为什么不是所有动态分析都要交给 AI”？
答：我会说 AI 在这里不是做真实动态执行，而是做“复核”和“解释”。真正的动态分析需要跑代码、跑构建、跑运行时环境，这个成本很高，也不一定安全。我的方案是先用静态分析尽量找确定性证据，再用 AI 处理那些非标准、低置信、需要上下文判断的部分。这样更现实，也更工程化。

## 十四、面试时的表达建议

### 131. 如果面试官问得很细，你回答的顺序应该是什么？
答：我建议按“先讲目标，再讲分层，再讲关键难点，再讲具体实现”来答。不要一上来就抛名词。比如先说“这个项目是为了解决依赖治理问题”，再说“我把它分成静态分析、证据链、signals、AI 复核几层”，这样面试官更容易跟上。

### 132. 如果面试官质疑“这些是不是 AI 做的，不是你自己做的”，你怎么回答？
答：我会强调：AI 只是增强层，底层 analyzer、evidence、signals、query/domain、monorepo 处理、输出整合这些都是工程实现。换句话说，AI 不是替代我写代码，而是基于我自己搭好的分析系统做复核和交互。这样边界就很清楚。

### 133. 如果面试官问“你这个项目最能体现你能力的地方是什么”，你怎么答？
答：我会说最能体现我能力的不是单点功能，而是我把一个原本偏工具型、偏单功能的项目，逐步做成了一个有分层架构、有证据链、有 monorepo 支持、有 AI 复核、有交互式 review 的系统。这个过程里涉及静态分析、工程架构、性能和成本权衡、提示词约束、交互设计等多个维度。

### 134. 如果面试官问“你这个项目最有技术含量的模块是什么”，你怎么答？
答：我会选 signals + candidate 分级 + AI 二次复核这一段。因为这段最能体现“传统静态分析如何和 AI 结合”，而且不是简单接个模型，而是先做结构化证据，再做分级，再控制成本，最后才引入 AI 做复核。

### 135. 如果面试官问“你这个项目后面还可以怎么演进”，你怎么答？
答：我会说有几条方向。第一是补更完整的 config/script/tooling 规则，让更多误报直接在静态层被修正。第二是把局部代码上下文进一步升级成真正的局部 RAG。第三是把 review 做成更强的 code review/治理助手，比如生成治理计划、风险排序、修复建议。第四是做历史 diff 和趋势分析，让它从一次性分析变成持续治理工具。

## 最后提醒

这份题库的正确使用方式不是逐字硬背，而是：
- 先记住每个模块的核心逻辑
- 再记住 2 到 3 个你最熟的典型例子
- 最后把回答讲成“我为什么这么做”

如果你能把下面这句话讲顺，面试基本就不会跑偏：

“这个项目的核心思路是：先用静态分析提取确定性事实，再整理成 evidence 证据链；对于传统方法拿不准的非常规依赖，我通过 signals 做线索补充，再用 candidate 分级控制范围，最后只让 AI 去复核那些低置信对象。这样既保留了工程上的稳定性，又补上了复杂场景下的解释和判断能力。”
